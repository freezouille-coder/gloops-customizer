🗺️ Zones Monde

Comment delimiter les zones de gameplay dans Maya pour le mode jeu 3eme personne.

1. 💡 Concept

Les zones de gameplay (pieton, route, parking, spawn...) sont definies dans Maya comme des meshes low-poly nommes, groupes sous ZONES.

Au runtime, le code parse les noms, rend les meshes invisibles, et utilise leur geometrie pour des tests point-in-polygon.

2. 📋 Convention de nommage

Prefixe meshType de zoneComportement in-game
zone_pedestrian_*Trottoir / chemin pietonNPCs wanderent ici, voitures interdites
zone_drive_right_*Route sens droitVoitures traffic suivent ce path (sens horaire)
zone_drive_left_*Route sens gaucheVoitures traffic sens anti-horaire
zone_parking_*ParkingVoiture se gare / spawn
zone_spawn_*Spawn generique NPCEndroit ou les NPCs apparaissent
zone_spawn_aveggies_*Spawn faction AveggiesSpawn biaise vers Bobo, Mush, NionNion
zone_spawn_sugarheads_*Spawn faction SugarHeadsSpawn biaise vers SugarHead, CryptoBro, ZogZog
zone_building_*Emplacement batimentZone interactive (F pour interagir)
zone_no_go_*Barriere invisibleBloque joueur + vehicules
zone_water_*Eau / merShader eau, collision splash
zone_runway_*Piste d'avionDecollage / atterrissage autorise
zone_helipad_*HeliportSpawn / landing helicoptere

3. 🎨 Format dans Maya

  • Meshes planes ou polygones aplatis sur le sol (la forme 2D vue du dessus = la zone)
  • Y = hauteur du sol dans cette zone
  • Pas de materiau visible — rendu invisible au runtime
  • Si une zone a du volume (tunnel, pont), utilise un cube
  • Grouper sous ZONES
  • Export en FBX separe du decor visible → fbx/WORLD/zones.fbx

Structure recommandee

ZONES
├── zone_pedestrian_01
├── zone_pedestrian_02
├── zone_drive_right_main
├── zone_drive_left_main
├── zone_building_garden
├── zone_building_candy_factory
├── zone_spawn_aveggies_forest
├── zone_spawn_sugarheads_plaza
├── zone_no_go_cliff
└── zone_helipad_rooftop

4. 🚦 Carrefours & feux rouges

Principe : un carrefour est une zone avec des feux qui cyclent. Les voitures freinent au rouge.

Nommage Maya

NomTypeRole
zone_intersection_01Plane au solCouvre toute l'intersection. Les voitures qui y entrent checkent le feu.
trafficlight_north_01LocatorPosition du feu pour les voitures venant du Nord
trafficlight_south_01LocatorPosition du feu pour les voitures venant du Sud
trafficlight_east_01LocatorPosition du feu pour les voitures venant de l'Est
trafficlight_west_01LocatorPosition du feu pour les voitures venant de l'Ouest

Le suffixe _01 doit matcher entre zone_intersection et les trafficlight_*.

Cycle par defaut

t = 0s     N/S vert       E/W rouge
t = 25s    N/S jaune      E/W rouge
t = 30s    N/S rouge      E/W vert
t = 55s    N/S rouge      E/W jaune
t = 60s    (boucle)

Cycle configurable par intersection dans config/world-entities.json :

"intersections": {
    "01": {
        "cycle_ns": 30,
        "cycle_ew": 30,
        "yellow_duration": 5,
        "start_phase": "ns_green"
    }
}

Les feux physiques

Au runtime, a chaque position trafficlight_*, un mesh "poteau + 3 lumieres" est instancie. Les 3 lumieres (rouge, jaune, vert) sont des spheres avec emissive que le code pilote selon l'etat du cycle.

5. 🛤️ Trottoirs (sidewalks)

Une voiture peut monter sur un trottoir si la geometrie le permet. Le code de collision utilise raycast + bbox, donc il respecte exactement ce que tu modelises.

3 approches de modeling

Tu veux...Modele ainsiComportement
Trottoir infranchissable Bord vertical a 90°, hauteur 15-30 cm La voiture bloque contre le mur. Collision nette.
Trottoir franchissable Bord incline 15-30°, ou hauteur faible (5-10 cm) Les roues peuvent monter en vitesse. Cool pour les poursuites / carambolages.
Trottoir visuel safe Bord vertical + zone_no_go_* invisible au-dessus Impossible de rouler dessus meme avec une rampe. Bon pour les zones sensibles.

Zone vs collision physique

⚠️ Attention a ne pas confondre :

  • zone_pedestrian_* = zone logique ou les NPCs piétons wanderent. Pas de barriere physique pour le joueur.
  • La collision du trottoir vient du mesh visible (le decor 3D)
  • Si tu veux un blocage imaterial → utilise zone_no_go_* en plus

6. 🛣️ Routes courbes (NURBS curves)

Pour des routes avec des virages, tu peux ajouter des NURBS curves nommees :

  • path_drive_right — spline que les voitures suivent en sens horaire
  • path_drive_left — sens anti-horaire
  • path_parade_* — trajet d'un evenement (parade, defile)

Au runtime, elles seront converties en THREE.CatmullRomCurve3 pour le traffic AI.

7. 🔀 Zones empilees (optionnel)

Si tu as besoin de plusieurs couches de zones au meme endroit (ex: route en dessous d'un pont pieton), utilise des cubes volumetriques au lieu de planes :

  • Un test point-in-polygon ne suffit plus — on teste l'inclusion dans un bbox 3D
  • La hauteur du joueur determine quelle zone est active

Pour la V1, reste sur des planes 2D (plus simple, plus performant).

8. ✅ Checklist

  1. Creer des planes/polygones nommes zone_<type>_<nom>
  2. Les poser a meme le sol de l'ile (Y = hauteur du sol)
  3. Les grouper sous un group ZONES
  4. (Optionnel) Ajouter des NURBS curves path_* pour les routes courbes
  5. Exporter en FBX separe → fbx/WORLD/zones.fbx
  6. Au runtime, le code les parse par nom, les rend invisibles, et utilise la geometrie pour les tests d'inclusion
